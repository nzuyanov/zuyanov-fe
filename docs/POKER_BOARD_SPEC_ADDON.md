## ЗАМЕНА: Секция 3.3 «Блайнды» → Автоматический расчёт блайндов

### Что меняется

Секция 3.3 из основного SPEC (ручной ввод SB, BB, интервала, множителя) **полностью заменяется** на автоматический расчёт. Пользователь больше не вводит блайнды вручную — вместо этого выбирает скорость игры из трёх пресетов.

### Новая секция «Блайнды» в модалке настроек

Вместо четырёх полей ввода (SB, BB, интервал, множитель) — **три кнопки-переключателя**:

```
Скорость игры:   [ 🐢 Медленная ]   [ ⚡ Стандартная ]   [ 🚀 Быстрая ]
```

| Скорость | Длительность уровня | Стартовый BB | Характер |
|----------|--------------------:|-------------:|----------|
| 🐢 Медленная | 20 мин | startingStack / 100 | Глубокая игра, постфлоп-покер, 100 BB на старте |
| ⚡ Стандартная | 15 мин | startingStack / 50 | Сбалансированная, классика домашних турниров, 50 BB на старте |
| 🚀 Быстрая | 10 мин | startingStack / 25 | Агрессивная, быстрые решения, 25 BB на старте |

По умолчанию выбрана «⚡ Стандартная».

### Алгоритм генерации таблицы блайндов

Алгоритм основан на стандартах домашних покерных турниров из PokerSoup и HomePokerTourney.

**Входные данные:**
- `startingStack` — стартовый стек игрока (в фишках)
- `playerCount` — количество игроков
- `gameDurationMinutes` — длительность турнира
- `speed` — выбранная скорость (`slow` | `standard` | `fast`)
- `chipDenominations` — список номиналов фишек из секции «Фишки» (для округления)

**Шаг 1: Определить параметры скорости**

```typescript
const speedParams = {
  slow:     { levelMinutes: 20, startingBBRatio: 100 },
  standard: { levelMinutes: 15, startingBBRatio: 50 },
  fast:     { levelMinutes: 10, startingBBRatio: 25 },
}
```

**Шаг 2: Рассчитать количество уровней**

```typescript
const totalLevels = Math.ceil(gameDurationMinutes / levelMinutes)
// + 2 запасных уровня на случай если турнир затянется
const levelsWithBuffer = totalLevels + 2
```

**Шаг 3: Рассчитать начальный и конечный BB**

Начальный BB:
```typescript
const startBB = roundToChip(startingStack / startingBBRatio, chipDenominations)
```

Конечный BB (уровень, на котором турнир должен завершиться):
Используем правило из покерной теории: турнир заканчивается, когда BB ≈ 5–10% от общего количества фишек в игре.

```typescript
const totalChips = startingStack * playerCount
const finalBB = roundToChip(totalChips * 0.05, chipDenominations)
```

**Шаг 4: Построить кривую роста**

Блайнды растут по геометрической прогрессии от `startBB` до `finalBB` за `totalLevels` уровней. Каждый следующий уровень ≈ в 1.5× больше предыдущего (оптимальный рост по PokerSoup — постоянный процент прироста ~33–50% на уровень).

```typescript
// Рассчитываем множитель для плавной кривой
const growthRate = Math.pow(finalBB / startBB, 1 / (totalLevels - 1))

const blindLevels: BlindLevel[] = []
for (let i = 0; i < levelsWithBuffer; i++) {
  const rawBB = startBB * Math.pow(growthRate, i)
  const bb = roundToChip(rawBB, chipDenominations)
  const sb = roundToChip(bb / 2, chipDenominations)

  blindLevels.push({
    level: i + 1,
    smallBlind: sb,
    bigBlind: bb,
    durationMinutes: levelMinutes,
  })
}
```

**Шаг 5: Округление до номинала фишки**

Все значения блайндов округляются ВВЕРХ до ближайшего доступного номинала фишки:

```typescript
const roundToChip = (value: number, denominations: number[]): number => {
  const sorted = [...denominations].sort((a, b) => a - b)
  const minChip = sorted[0]

  // Округляем до ближайшего кратного минимальному номиналу (вверх)
  const rounded = Math.ceil(value / minChip) * minChip

  // Убеждаемся что значение не меньше минимального номинала
  return Math.max(rounded, minChip)
}
```

**Шаг 6: Убрать дубликаты**

Если два соседних уровня дали одинаковый BB после округления — убрать дубликат (оставить один уровень).

## 13. Минимальный рейз — отображение на борде

### 13.1 Зачем

В домашних турнирах игроки регулярно путают минимальный размер рейза. Споры на тему «а сколько минимум я могу поднять?» — одна из самых частых пауз в игре. Отображение этого значения на борде устраняет проблему полностью.

### 13.2 Правила NL Hold'em

По правилам No-Limit Texas Hold'em:
- **Минимальный бет** (первая ставка в раунде) = размер BB
- **Минимальный рейз** = предыдущая ставка + размер последнего повышения
- На префлопе: минимальный рейз = BB + BB = 2 × BB (поскольку BB считается «ставкой», а минимальное повышение = размеру BB)

### 13.3 Что показывать на борде

Для домашних турниров достаточно упрощённого отображения:

```
Мин. бет: 100     Мин. рейз: 200
```

Где:
- **Мин. бет** = текущий BB (минимальная ставка, если до тебя никто не ставил)
- **Мин. рейз** = 2 × BB (минимальное повышение на префлопе)

Эти значения **обновляются автоматически** при смене уровня блайндов.

### 13.4 Расположение

В блоке «Блайнды» на информационной панели (правая зона), сразу под текущими блайндами:

```
БЛАЙНДЫ: 50 / 100
Мин. рейз: 200
След. уровень: 100 / 200
```

### 13.5 Стиль

- Шрифт мельче, чем у блайндов, но крупнее чем у «след. уровень»
- Цвет: `--text-secondary` (приглушённый, не отвлекает от основных блайндов)
- Моноширинный, как все числа

### 13.6 Типы

Добавить геттер в стор:

```typescript
// в usePokerStore
minBet: computed(() => currentBlinds.value.bb)
minRaise: computed(() => currentBlinds.value.bb * 2)
```

## 14. Средний стек в BB

### 14.1 Зачем

Средний стек в больших блайндах (BB) — ключевая метрика турнира. Она показывает, насколько «глубокая» игра осталась и на какой стадии находится турнир:
- **40+ BB** — глубокая игра, можно играть постфлоп
- **20–40 BB** — средняя глубина, важны префлоп-решения
- **10–20 BB** — короткие стеки, push-or-fold территория
- **< 10 BB** — критически короткие стеки

Для домашнего турнира это наглядный индикатор того, насколько «напряжённая» сейчас ситуация.

### 14.2 Формула

```
averageStackBB = (startingStack × totalPlayers) / activePlayers / currentBB
```

Где:
- `startingStack × totalPlayers` = общее количество фишек в игре (не меняется, фишки выбывших остаются в игре)
- `activePlayers` = количество активных (невыбывших) игроков
- `currentBB` = текущий большой блайнд

**Примечание:** если в турнире есть ребаи, то общее количество фишек увеличивается. Формула с учётом ребаев:

```
totalChips = (startingStack × totalPlayers) + (totalRebuys × rebuyChips) + (totalAddOns × addOnChips)
averageStackBB = totalChips / activePlayers / currentBB
```

Отображать округлённо до целого числа.

### 14.3 Расположение

В блоке статистики на информационной панели, рядом с банком:

```
БАНК: 15 000 ₽
Игроков: 5 / 8
Ср. стек: ~42 BB
```

### 14.4 Стиль

- Значение: моноширинный шрифт, цвет зависит от диапазона:
  - `> 40 BB` → зелёный (`--color-success` / покерный зелёный)
  - `20–40 BB` → золотой (вторичный акцент)
  - `10–20 BB` → оранжевый (`--color-warning`)
  - `< 10 BB` → красный (`--color-error`)

### 14.5 Типы

```typescript
// в usePokerStore — новый геттер
averageStackBB: computed(() => {
  const totalChips = config.startingStack * config.playerCount
    + gameState.totalRebuys * config.rebuyChips
    + gameState.totalAddOns * config.addOnChips
  const active = activePlayers.value.length
  if (active === 0 || currentBlinds.value.bb === 0) return 0
  return Math.round(totalChips / active / currentBlinds.value.bb)
})
```

## 15. Номер раздачи на борде

### 15.1 Зачем

Номер раздачи (hand number) помогает отслеживать прогресс игры и полезен при обсуждении спорных ситуаций после турнира («помнишь в 47-й раздаче?..»).

## 16. Индикация стадии турнира

### 16.1 Зачем

В покерных турнирах стратегия кардинально меняется в зависимости от стадии. Визуальная индикация помогает игрокам осознавать текущую ситуацию и добавляет атмосферу.

### 16.2 Определение стадий

Стадия рассчитывается автоматически по двум факторам: доля оставшихся игроков и средний стек в BB.

| Стадия | Условие | Цвет бейджа |
|--------|---------|-------------|
| **Ранняя** | > 70% игроков в игре | зелёный |
| **Средняя** | 40–70% игроков в игре | золотой |
| **Баббл** | Осталось на 1 больше, чем призовых мест (напр. 4 игрока при 3 призовых) | красный, мигающий |
| **В призах** | Игроков ≤ призовых мест, но > 3 | синий |
| **Финальный стол** | ≤ 3 игрока | золотой, особый стиль |
| **Хедз-ап** | 2 игрока | красный, особый стиль |

### 16.3 Определение баббла

Баббл — момент, когда осталось на 1 игрока больше, чем призовых мест. Для стандартного домашнего турнира с 3 призовыми местами:
- Баббл = 4 активных игрока
- Лопнул баббл = 3 активных игрока (все в призах)

Формула:
```
isBubble = activePlayers === prizeDistribution.places.length + 1
```

### 16.4 Расположение

Бейдж стадии — в хедере или в верхней части информационной панели:

```
[РАННЯЯ СТАДИЯ]  или  [🔴 БАББЛ!]  или  [🏆 ФИНАЛЬНЫЙ СТОЛ]
```

### 16.5 Стиль

- Бейдж: скруглённый, с фоном соответствующего цвета
- На стадии «Баббл»: пульсирующая анимация, чтобы привлечь внимание
- На стадии «Финальный стол»: золотой бейдж с иконкой трофея

### 16.6 Типы

```typescript
type TournamentStage = 'early' | 'middle' | 'bubble' | 'in-prizes' | 'final-table' | 'heads-up'

// в usePokerStore — новый геттер
tournamentStage: computed((): TournamentStage => {
  const active = activePlayers.value.length
  const total = config.playerCount
  const prizeCount = config.prizeDistribution.places.length

  if (active <= 2) return 'heads-up'
  if (active <= 3) return 'final-table'  // для домашнего турнира 5-8 чел
  if (active <= prizeCount) return 'in-prizes'
  if (active === prizeCount + 1) return 'bubble'
  if (active / total > 0.7) return 'early'
  return 'middle'
})
```

### 16.7 Звуковые события (дополнение к секции 7 SPEC)

| Событие | Звук |
|---------|------|
| Наступил баббл | Особый сигнал (тройной тон) |
| Баббл лопнул (все в призах) | Победный аккорд |

## 17. Превью всех уровней блайндов

### 17.1 Зачем

В текущем SPEC (секция 3.3, Блайнды) предлагается показывать «превью таблицы уровней (первые 5–8 уровней)». Для турнира важно видеть **все** уровни, чтобы:
- Понять, на каком уровне блайнды станут критически высокими
- Оценить, до какого уровня хватит текущего стека
- Оператор может скорректировать множитель до старта, если уровни выглядят неправильно

### 17.2 В настройках (модалка)

Заменить «первые 5–8 уровней» на полную таблицу с прокруткой:

```
┌───────┬─────┬──────┬────────────┐
│ Ур.   │ SB  │  BB  │  Мин. рейз │
├───────┼─────┼──────┼────────────┤
│  1    │  25 │   50 │    100     │
│  2    │  50 │  100 │    200     │
│  3    │ 100 │  200 │    400     │
│  4    │ 200 │  400 │    800     │
│  5    │ 300 │  600 │   1200     │
│  ...  │ ... │  ... │    ...     │
│ 12    │ ... │  ... │    ...     │  ← последний уровень
└───────┴─────┴──────┴────────────┘
```

Особенности:
- Таблица скроллируемая вертикально (это единственное место, где допустим скролл — в модалке настроек)
- Количество уровней = `Math.ceil(gameDurationMinutes / blindIntervalMinutes)`
- Каждый уровень показывает: номер, SB, BB, мин. рейз (= 2 × BB)
- Текущий уровень блайндов подсвечивается (во время игры, если открыта модалка для просмотра)
- Если множитель/начальные значения изменились — таблица пересчитывается в реальном времени

### 17.3 На борде (во время игры)

На информационной панели — не полная таблица, а **компактное превью** следующих 2–3 уровней:

```
БЛАЙНДЫ: 100 / 200        ← текущие (крупно)
Мин. рейз: 400
До смены: 08:32

→ 200 / 400               ← следующие (мелко)
→ 400 / 800
```

Это уже частично покрыто в SPEC («Следующий уровень: SB / BB»), но расширяем до 2–3 следующих уровней.

### 17.4 Доступ к полной таблице во время игры

Опциональная кнопка (иконка таблицы / «📋») на информационной панели:
- При нажатии: модалка с полной таблицей уровней
- Текущий уровень подсвечен
- Пройденные уровни затемнены
- Таймеры НЕ останавливаются (это не пауза, просто справочная инфо)

## 21. Автоматическая раздача фишек из чемодана

### 21.1 Зачем

Перед каждым домашним турниром организатор тратит время на подсчёт: «У нас 300 белых фишек по 25, 200 красных по 50... сколько каждому дать?» Приложение должно делать это автоматически — на основе содержимого чемодана, количества игроков и стартового стека.

### 21.2 Что вводит пользователь (секция «Фишки» в модалке)

Пользователь описывает **содержимое чемодана** — полный набор фишек, которые физически есть:

| Поле | Тип | Описание | Пример |
|------|-----|----------|--------|
| Номинал | `number` | Стоимость фишки | 25 |
| Цвет | `string` (опционально) | Для наглядности | Белый |
| Количество в чемодане | `number` | Сколько физически есть фишек этого номинала | 300 |

Кнопки «+ Добавить номинал» / «Удалить». Минимум 1 номинал.

### 21.3 Алгоритм автораздачи

**Входные данные:**
- `chipCase` — массив `{denomination, count}` (содержимое чемодана)
- `playerCount` — количество игроков
- `startingStack` — стоимость стартового стека в фишках
- `maxRebuys` — макс. ребаев на игрока
- `addOnEnabled` — есть ли аддон

**Задача:** распределить фишки из чемодана так, чтобы:
1. Каждый игрок получил набор фишек на сумму = `startingStack`
2. Фишек в чемодане хватило на всех игроков + запас на ребаи/аддоны
3. Мелких номиналов было больше (для удобства ставок на ранних уровнях)
4. Крупных номиналов было меньше (они понадобятся позже)

**Алгоритм:**

```typescript
const calculateChipDistribution = (
  chipCase: ChipCaseEntry[],
  playerCount: number,
  startingStack: number,
  maxRebuys: number,
  addOnEnabled: boolean
): ChipDistribution => {
  const sorted = [...chipCase].sort((a, b) => a.denomination - b.denomination)

  // Шаг 1: Рассчитать сколько раздач нужно (старт + ребаи + аддоны)
  // Оценка: не все игроки используют все ребаи, берём ~60% от максимума
  const estimatedRebuys = Math.ceil(playerCount * maxRebuys * 0.6)
  const estimatedAddOns = addOnEnabled ? playerCount : 0
  const totalDistributions = playerCount + estimatedRebuys + estimatedAddOns

  // Шаг 2: Определить доступное количество на одну раздачу
  const availablePerDistribution = sorted.map(chip => ({
    denomination: chip.denomination,
    maxPerPlayer: Math.floor(chip.count / totalDistributions),
  }))

  // Шаг 3: Жадный алгоритм — набираем стек начиная с мелких фишек
  // Приоритет: мелких больше (нужны для блайндов), крупных меньше
  let remaining = startingStack
  const distribution: { denomination: number; count: number }[] = []

  // Проход от крупных к мелким — определяем минимум крупных
  for (let i = sorted.length - 1; i >= 0; i--) {
    const chip = availablePerDistribution[i]
    if (chip.denomination > remaining) continue

    if (i === 0) {
      // Последний (самый мелкий) — берёт весь остаток
      const count = Math.ceil(remaining / chip.denomination)
      distribution.unshift({ denomination: chip.denomination, count })
      remaining -= count * chip.denomination
    } else {
      // Крупные фишки — берём умеренно (20-30% от доступного, минимум 1)
      const maxByValue = Math.floor(remaining / chip.denomination)
      const moderate = Math.max(1, Math.min(
        Math.ceil(chip.maxPerPlayer * 0.25),
        maxByValue
      ))
      distribution.unshift({ denomination: chip.denomination, count: moderate })
      remaining -= moderate * chip.denomination
    }
  }

  // Шаг 4: Корректировка — если набрали слишком много, убрать лишнее
  // Если набрали слишком мало — добавить мелких
  // ... (подробная балансировка)

  return {
    perPlayer: distribution,
    totalValue: distribution.reduce((s, d) => s + d.denomination * d.count, 0),
    totalChips: distribution.reduce((s, d) => s + d.count, 0),
    isValid: remaining === 0, // true если точно набрали стек
  }
}
```

### 21.4 Отображение в модалке настроек

Под списком номиналов чемодана — блок **«Раздача на игрока»**:

```
📦 Чемодан: 300×25 | 200×50 | 100×100 | 50×500

👤 Каждый игрок получает (стек 1500):
   8 × 25  = 200
   8 × 50  = 400
   5 × 100 = 500
   1 × 500 = 500  ←— если не хватает без крупных, добавляет
   ──────────────
   Итого: 22 фишки = 1 600  ⚠️ (не точно 1500 — ближайшее возможное)

📊 Хватит на: 6 игроков + ~4 ребая + 6 аддонов
⚠️ Не хватит на все ребаи (нужно 12, доступно ~10)
```

**Состояния:**
- ✅ Зелёный — фишек хватает на всё (старт + макс. ребаи + аддоны)
- ⚠️ Жёлтый — фишек хватает на старт, но может не хватить на все ребаи
- ❌ Красный — фишек не хватает даже на стартовую раздачу всем игрокам

### 21.5 Что если стек нельзя набрать точно

Если стартовый стек (напр. 1500) нельзя набрать точно из доступных номиналов (напр. номиналы 25, 100, 500 → нельзя набрать ровно 1500 из 25 и 100 без 500):
- Алгоритм подбирает **ближайшее большее** значение
- Показывает предупреждение: «Точный стек 1500 невозможен. Ближайший: 1525 (+25)»
- Пользователь может скорректировать стартовый стек или номиналы

## 22. ЗАМЕНА: Правила ребая — когда можно докупаться

### 22.1 Текущее состояние в SPEC

В текущем SPEC (секция 5.3) правила ребая описаны нечётко — не указан порог, при котором ребай доступен.

### 22.2 Стандартные правила покерных турниров

По стандартным правилам покерных турниров существуют два формата ребая:

**Формат 1: «Ребай при нуле» (Bust-out rebuy)**
Ребай доступен только когда игрок потерял все фишки. Фактически это re-entry, но без смены места.

**Формат 2: «Ребай при стеке ≤ стартового» (Standard rebuy)**
Ребай доступен когда стек игрока равен или меньше стартового стека. Например, если стартовый стек 10 000, можно докупиться при 9 999 фишках, при 10 000, но нельзя при 10 001.

Если игрок потерял все фишки, он может купить сразу два стека.

### 22.3 Рекомендация для домашнего турнира

Для домашнего формата (5–8 игроков, один стол, оператор-ведущий) используем **упрощённый вариант**:

**Ребай доступен когда игрок потерял ВСЕ фишки** (формат 1 — bust-out rebuy).

Причины:
- В домашнем турнире невозможно точно отслеживать стек каждого игрока (приложение не знает, сколько фишек у кого на руках — это не онлайн-покер)
- Факт «у игрока кончились фишки» очевиден для всех за столом
- Это проще для оператора: нажал «Ребай» → игроку выдали стандартный набор фишек

### 22.4 Изменения в секции 4.3 (карточка игрока) и 5.3 (ребаи)

Кнопка «+ Ребай» на карточке игрока:
- **Видна** только в период ребаев И если у игрока остались ребаи
- При нажатии: модалка «Игрок X докупается? Получит стандартный набор фишек (стек 1500). Ребай N из M.»
- **Действие**: банк увеличивается на buy-in, счётчик ребаев обновляется, призовые пересчитываются

**Аддон** (после окончания периода ребаев):
- В отличие от ребаев, аддон доступен ЛЮБОМУ игроку независимо от размера стека — это стандартное правило
- Аддон = 1 раз, стоимость = buy-in, фишки = стартовый стек
- Кнопка «Add-on» появляется у ВСЕХ активных игроков после окончания ребай-периода

### 22.5 Что получает игрок при ребае / аддоне

Игрок получает **тот же набор фишек**, что и на старте — рассчитанный алгоритмом из секции 21.3. Это важно: оператор видит на экране точный состав набора и физически выдаёт фишки:

```
Ребай Игрок 3:
Выдать: 8×25, 8×50, 5×100, 1×500
```

Эта информация должна показываться в модалке подтверждения ребая.
